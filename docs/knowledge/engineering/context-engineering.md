# Context Engineering

Context Engineering 是围绕上下文窗口做信息组织、压缩、检索和注入的工程能力。对 Agent 来说，它决定模型在每一步能看到什么、记住什么、忽略什么，以及如何把外部工具和历史轨迹转成可用上下文。

它和 Prompt Engineering 的区别是：

- Prompt Engineering 更关注单次输入如何写。
- Context Engineering 更关注多源信息如何动态进入上下文。

## 1. 为什么重要

Agent 失败经常不是模型完全不会，而是上下文给错了：

- 检索结果太多，关键信息被淹没。
- 历史轨迹太长，模型忘了当前目标。
- 工具结果格式混乱，模型误读字段。
- 多轮对话没有压缩，窗口被无关内容占满。
- 代码仓库或文档检索没有结构，定位不到关键文件。

## 2. 上下文来源

| 来源 | 示例 | 风险 |
| --- | --- | --- |
| 用户输入 | 当前问题、约束、偏好 | 需求模糊、提示注入 |
| 系统指令 | 角色、规则、安全边界 | 过长、冲突、不可维护 |
| 对话历史 | 多轮上下文、澄清信息 | 噪声累积、遗忘重点 |
| RAG 结果 | 文档片段、网页、知识库 | 召回错、排序错、过期 |
| 工具结果 | API 返回、代码执行、搜索结果 | 字段误读、异常未处理 |
| Agent 状态 | 当前计划、已完成步骤、待办 | 状态不一致、循环 |
| 长期记忆 | 用户画像、项目偏好、历史经验 | 隐私、污染、过拟合 |

## 3. 关键技术

### 3.1 信息分层

上下文应按稳定性和优先级分层：

1. System policy：不能被用户覆盖的规则。
2. Task state：当前目标、约束、成功标准。
3. Working memory：最近几步计划、观察和决策。
4. Retrieved evidence：与当前步骤相关的外部证据。
5. Historical memory：必要时才注入的长期信息。

### 3.2 压缩与摘要

长任务需要持续压缩上下文：

- 滚动摘要：把历史对话压缩成当前状态。
- 结构化摘要：按目标、已知事实、待办、风险拆分。
- 工具结果摘要：保留关键字段和错误码，删除冗余。
- 轨迹摘要：保留决策理由，不保留所有 token。

### 3.3 检索增强

RAG 不只是向量检索：

- Query rewrite：把用户问题改写成更适合检索的查询。
- Hybrid search：BM25 + 向量召回。
- Rerank：用重排模型筛选更相关片段。
- Context packing：按证据重要性和 token 预算组织片段。
- Citation：让最终答案能追踪到证据。

### 3.4 上下文防污染

Agent 要防止外部内容覆盖系统规则：

- 检索结果和网页内容必须标记为 untrusted content。
- 工具输出只当数据，不当指令。
- 用户上传文档不能修改工具权限。
- 对 prompt injection 样本做离线评测。

## 4. Agent 场景设计

### 4.1 Coding Agent

上下文应包含：

- 仓库结构。
- 相关文件片段。
- 测试失败日志。
- 约束和编码规范。
- 最近修改 diff。

避免一次性塞入全仓库。更好的做法是：搜索 -> 打开相关文件 -> 局部编辑 -> 跑测试 -> 根据失败继续检索。

### 4.2 企业知识库 Agent

上下文应包含：

- 用户问题。
- 权限过滤后的知识片段。
- 文档发布时间和版本。
- 冲突信息提示。
- 引用来源。

高风险问题要返回“无法从授权知识库确认”，而不是编造。

### 4.3 多工具 Agent

上下文应包含：

- 当前任务状态。
- 可用工具 schema。
- 上一步工具结果。
- 错误码和重试策略。
- 哪些动作需要人工确认。

工具列表不应无限扩张，常用工具可以按任务动态裁剪。

## 5. 评测指标

| 指标 | 含义 |
| --- | --- |
| Retrieval precision | 注入上下文中真正相关的比例 |
| Retrieval recall | 关键证据是否被召回 |
| Context utilization | 模型是否正确使用已给证据 |
| Lost-in-the-middle rate | 中间位置证据是否被忽略 |
| Injection robustness | 面对恶意上下文是否仍遵守规则 |
| Token efficiency | 单次任务消耗的上下文 token |

## 6. 面试要点

1. Context Engineering 和 Prompt Engineering 有什么区别？
2. 长对话 Agent 如何管理上下文？
3. RAG 检索结果太多时如何筛选和压缩？
4. 如何避免工具结果或网页内容污染系统指令？
5. Coding Agent 如何构造仓库上下文？
6. 如何评估模型是否真的使用了检索证据？

## 7. 实践 Checklist

- 每个上下文片段都带来源、时间、可信级别。
- 对工具输出、检索内容、用户文档做不可信标记。
- 长任务必须有结构化 state，而不是只靠聊天历史。
- 上下文注入前做 token budget 分配。
- 对关键事实要求引用或工具证据。
- 定期用 bad case 更新检索和上下文打包策略。
