## 完整面试题列表（26 道）

### 一、项目经验（5 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 1 | 做个自我介绍，重点介绍Agent相关项目经历 | 突出 Agent 相关的技术栈（LangChain/LangGraph/Coze）、落地效果、解决的核心问题 |
| 2 | 你的Agent项目整体架构是什么？核心模块有哪些？ | 路由层→意图识别→任务拆解→工具调用循环→结果聚合；核心模块：推理引擎、工具管理、记忆系统、监控日志 |
| 3 | 项目中用了哪些框架（如LangChain、Coze）？为什么选这个框架？优缺点是什么？ | LangChain：灵活但重、生态丰富但版本迭代快；Coze：拖拽式编排上手快但可定制性弱；LangGraph：适合有状态复杂工作流 |
| 4 | 项目最大的技术挑战是什么？怎么解决的？有什么落地效果？ | 结合具体项目讲，如上下文溢出→分层压缩、工具调用失败→重试降级策略、幻觉→RAG增强+校验 |
| 5 | 你提到的Coze，拖拽式编排与Agent原生模式有何区别？你如何设计节点？ | 拖拽式=可视化工作流（适合简单场景）；原生=代码定义Agent循环（适合复杂逻辑）；节点设计：输入/输出Schema明确、异常处理前置 |

### 二、Agent 核心范式（4 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 6 | 解释Agent的核心范式（ReAct、Plan-and-Execute、反应式等），区别与适用场景 | ReAct=边想边做（适合需外部交互的复杂任务）；Plan-and-Execute=先规划再执行（适合确定性要求高的任务）；反应式=直接响应（适合简单快速场景） |
| 7 | 什么是Agentic Loop（感知-规划-行动-记忆闭环）？各环节作用是什么？ | 感知：接收用户输入与环境反馈→规划：拆解任务选择策略→行动：调用工具执行操作→记忆：维护上下文与历史 |
| 8 | Agent如何防止死循环？有哪些终止条件设计？ | 最大迭代次数硬限制(通常10-15轮) + 循环模式检测(连续3次同一工具失败→放弃工具) + 终止条件：final_answer输出/超时/人工中断 |
| 9 | 上下文管理与记忆机制怎么做？Working/Short-Term/Long-Term记忆如何分层？ | Working Memory=当前任务上下文(in-context)；Short-Term=最近对话存Redis带过期时间；Long-Term=用户画像存向量库+需要时召回；分层索引(热7天/温30天/冷归档) |

### 三、Agent 关键技术（7 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 10 | 如何解决Agent的幻觉问题？有哪些校验与反思机制？ | RAG增强(用事实约束输出) + 置信度校验(要求模型标注不确定部分) + 事实核查(交叉验证) + Reflexion反思(记录失败轨迹并修正) |
| 11 | 什么是MCP？和Function Calling的区别？优缺点？ | MCP=标准协议(跨平台互操作、生态丰富、但学习成本高)；Function Calling=单一模型接口(简单易用、但不可移植、锁定特定模型) |
| 12 | Tool调用流程是什么？如何设计Tool的description以提升调用准确率？ | 分析需求→生成参数→执行→获取结果→反馈Observation→决定下一步；description设计：明确输入/输出Schema、写清楚功能边界、给使用示例、注明异常情况 |
| 13 | 工具调用失败如何处理？重试、降级、参数修正策略？ | 重试退避(Exponential Backoff) + 降级兜底(规则匹配/转人工) + 参数修正(让LLM重新生成参数) + 连续3次同一工具失败→放弃该工具 |
| 14 | RAG的核心流程？Chunking策略、Query Rewrite、Rerank怎么做？ | Chunking：按语义边界切片+10-20%重叠；Query Rewrite：扩展意图+分解子查询；Rerank：初筛Top-K后用Cross-Encoder精排压到3-5 |
| 15 | RAG检索内容与模型知识冲突时，如何处理？ | 优先信任检索结果(时效性更强) + 用置信度校验冲突 + 标注信息来源让用户判断 + 定期更新知识库 |
| 16 | 工具过多导致上下文过长，怎么优化？ | 按任务类型动态加载工具(只加载当前需要的) + 工具元数据摘要(不传完整Schema只传名称+简要描述) + 两步路由(先选工具类再选具体工具) |

### 四、Prompt 设计（3 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 17 | 设计Agent的System Prompt核心原则？包含哪些关键信息？ | 角色定义+约束规则+输出格式+异常处理指引+安全边界；关键信息：你是谁、你能做什么、你不能做什么、出错怎么办 |
| 18 | 如何用Few-shot、CoT提升输出稳定性？怎么避免格式混乱？ | 给2-3个示例引导格式 + CoT让模型先推理再输出 + 用Schema校验(JSON Schema强校验输出格式) + 明确说"不要输出XX格式" |
| 19 | Prompt如何做鲁棒性优化？反例提示、Schema校验怎么落地？ | 反例提示(明确列出不能做的事如"不要猜测数据") + Schema校验(输出后JSON解析校验) + 多轮自纠(输出→检查→修正循环) |

### 五、Multi-Agent（3 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 20 | Multi-Agent适用场景？为什么拆分多个Agent而非单Agent？ | 任务复杂度超过单Agent能力上限 + 需要并行处理 + 不同子任务需要不同专业能力 + 单Agent注意力漂移/推理链断裂 |
| 21 | 多Agent如何分工（角色设计）？协作流程与通信机制？ | 角色按能力域划分(编剧Agent/画师Agent/审核Agent) + 协作流程：任务分解→分配→执行→汇聚→审核 + 通信：共享消息队列/状态同步/事件驱动 |
| 22 | 如何避免多Agent幻觉放大？交叉验证与状态同步怎么做？ | 交叉验证(多个Agent互相校验输出一致性) + 状态同步(共享事实库避免信息不一致) + 仲裁机制(冲突时用高置信度Agent裁决) |

### 六、系统设计题（2 题）

| # | 面试题 | 参考要点 |
|---|--------|---------|
| 23 | 设计一个全自动化AI漫剧创作Agent，整体架构、核心节点、关键流程？ | 编剧Agent(剧情生成)→画师Agent(画面生成)→审核Agent(质量把关)；核心节点：角色库(RAG)、画风Prompt库、场景描述→分镜→画面→审核→修正循环；人物一致性：角色设定RAG+固定画风Seed |
| 24 | 漫剧Agent如何保证人物设定、画风前后一致？用什么技术方案？ | 角色设定存入向量库(每次生成前召回) + 画风用固定Seed+Prompt模板 + 关键帧一致性检测(Cosine Similarity) + 人工审核兜底 |
| 25 | 设计一个"可进化"的游戏BOSS战Agent，能学习玩家打法并动态调整策略 | 观测Agent(收集玩家行为序列)→策略Agent(RL动态调整)→进化Agent(跨回合学习)；Reward设计：击败玩家=正奖励、被击败=负奖励+策略更新；状态空间：玩家攻击模式/技能使用频率/回避策略 |
| 26 | 如何设计实验验证Agent效果（如漫剧质量、游戏AI体验），而非单纯提升难度？ | 漫剧：人物一致性率+画风匹配度+剧情连贯性评分(LLM-as-Judge) + 用户AB测试满意度；游戏：玩家胜率曲线+对战时长分布+策略多样性(不重复使用同一策略) |

## 面经总结

这是一场**高强度的Agent开发一面**，26道题覆盖Agent全栈知识：

- **项目经验**（5题）：重点考察项目落地能力和框架选型思考
- **核心范式**（4题）：ReAct/Plan-and-Execute/反应式+Agentic Loop
- **关键技术**（7题）：MCP/Function Calling/RAG/工具管理/幻觉防控
- **Prompt设计**（3题）：System Prompt/Few-shot/鲁棒性
- **Multi-Agent**（3题）：分工协作/幻觉防控
- **系统设计**（2题+扩展）：漫剧创作Agent/可进化游戏BOSS

面试特点：**不考算法题**，全考Agent工程实战，面试官对细节追问很深。
